User Guide to %lstattrib

Last updated: 28 Feb 2014

Introduction

The %lstattrib macro was written as a tool for use in a data pooling exercise. Its purpose was to list the full attributes of a correctly structured dataset to the log in a form that could be copied and pasted from the log and emailed to programmers who could then use the code in a final step to apply the correct attributes to the dataset(s) they were building. Then when all these datasets were brought together, the structure and attributes of the collection of these datasets would be correct.

Correct attributes includes keeping only the required variables, making sure the variables were in the correct order with the correct lengths and making sure all the correct formats, informats and variable labels were assigned.

Some programmers reading this might think this is easy because the characteristics of the first dataset in the list of datasets being "SET"ted together will pass on the attributes to all the following datasets. Unfortunately, this is only partly true. If all variable characteristics are assigned in the first dataset then these will be assigned to the other datasets but if some attributes are  intentionally left unassigned, such as labels, formats and informats, then these will be picked up from the first following dataset where these are given a value. And if the later datasets contain variables that do not exist in the first dataset then these variables will be added in the accumulated dataset. These extra variables might be the misspelled name of valid variables. In a pooling exercise, with data coming from many sources and where there are many variables, there is a lot of room for errors that will cause an error in the final structure. Hence, a tool is required to help make sure that all datasets being pooled have the same correct structure.

The macro can be linked to here.

The Evolution of %lstattrib

As stated in the above section, the tool was meant to be used interactively. The macro is run on a dataset which you are sure is fully correct, it writes the structure to the log, then you copy and paste what is written to the log and email it to the programmers. But the tool is interactive and can not be used programmatically. In the end, the macro needed to be used programmatically so many enhancements were made to the macro to allow this. To understand the macro and to know how to use it, we will need to retrace its usage from the way it used to work interactively through to the way it can be used programmatically. For programmatic use, a full User Guide is required which is the subject of this page. But the User Guide will only make sense if we go through the evolution of the macro.

The Beginnings

Here we see the macro being used to list the contents of sashelp.class to the log.  The macro was enhanced from the earliest version to add code to initialize all the variables. It now does this by default. So to reconstruct how the macro would have originally worked I have to tell it not to create the initialization code by setting the parameter init=no. The original version of the macro did not have this parameter.

You will see that the log output is neatly indented so that it can be copied and pasted as code. In the top code line you can see it has found that the dataset has a label 'Student Data' so it assign that. The LENGTH statement makes sure the variables are in the correct order with the correct length. Where the list of variables flows onto multiple lines, the macro will structure the list in a pleasing way.

The SET statement can be turned into a merge statement if required. This is where the input data is defined. It needs to go after the LENGTH statement and before where the formats and informats get cancelled out.

The attributes are assigned to the variables in alphabetical order, not the variable position order, to make it easier to find the variable in the list. If there are formats and informats then these will be displayed in a pleasing layout.

Lastly, the KEEP list names only those variables that were in sashelp.class so that if extra variables were introduced in the SET or MERGE then these extra variables will not be kept. Again, this is in alphabetical order to make it easier to find variable names. It is the LENGTH statement that defines the order of the variables.
 
12738  %lstattrib(sashelp.class,init=no);

******  Attributes obtained from sashelp.class  ******;
DATA xxxxxx(label='Student Data');
  *- The order of the variables in the following LENGTH statement matches -;
  *- the variable order in the original dataset so do not change.         -;
  LENGTH NAME $ 8 SEX $ 1 AGE 8 HEIGHT 8 WEIGHT 8
         ;

  *- The MERGE or SET statement for the input dataset(s) should go here. -;
  SET yyyyyy;

  *- Cancel existing formats and informats in the input dataset(s) -;
  FORMAT   _all_ ;
  INFORMAT _all_ ;

  *- Assign output variable attributes -;
  ATTRIB
    AGE    label=" "
    HEIGHT label=" "
    NAME   label=" "
    SEX    label=" "
    WEIGHT label=" "
    ;

  *- KEEP statement for the variables listed above -;
  KEEP AGE HEIGHT NAME SEX WEIGHT
       ;
RUN;

The Initialization Code

The next enhancement the macro received was for it to also create initialization code and this was made to be the default. The reason this was done was because it became a requirement to populate the variables and not just assume they were already in the input dataset(s). There might be a lot of these variables and if so then there is a chance of variable names being misspelled or numeric variables being given character values and vice versa. So the macro by default will list out all these variables and initialize them with a space or a missing value depending on whether they are numeric or character variables. The order used is the order in the output dataset as defined in the LENGTH statement. The reason for using this order if because "decodes" of variables often follow the coded form so it is better and more logical if these are kept in variable order.

         ;

  *- The MERGE or SET statement for the input dataset(s) should go here. -;
  SET yyyyyy;

  *- Overwrite the following missing values with what you are populating the  -;
  *- variables with. You may have to change the order of the variables where  -;
  *- there are dependencies such that the source variable is populated first. -;
  *- If you follow this method then when you get notes in the log about       -;
  *- uninitialised variables you will know that you are trying to populate a  -;
  *- variable with another variable that does not exist. You also avoid the   -;
  *- problem of spelling a variable name incorrectly when you assign a value  -;
  *- to it which can easily happen if there are a large number of variables.  -;

  SEX    = ' ';

  ATTRIB
    AGE    label=" "
    HEIGHT label=" "
    NAME   label=" "
    SEX    label=" "
    WEIGHT label=" "
    ;

  *- KEEP statement for the variables listed above -;
  KEEP AGE HEIGHT NAME SEX WEIGHT
       ;
RUN;

A Reminder Why we Need this Tool

Before we get into the topic of using the macro programmatically I would like to remind you why we need such a tool. Although the characteristics of the first dataset, where datasets are being SET'ted together, get passed onto the later datasets, then where attributes are not assigned, it will pick these attributes up from later datasets where they are assigned. Also, if variables are introduced in later datasets that were not there in the first datasets then these will be added to the resulting output dataset. These new variables might be an accidental misspelling of a valid variable. Only a tool such as %lstattrib can avoid this from happening.

Setting Out Down the Path of Programmatic Use

As stated near the beginning, %lstattrib was intended purely as an interactive tool to write sas code to the log that would ensure datasets had the correct structure. This code was then intended to be emailed to the other programmers to use so that the datasets they were pooling all had the correct structure. But then the need to use the tool in a programmatic way arose and so the macro needed to be enhanced to do this. The first thing to be changed was to enable the output to be routed to a location other than the log. The logfile= parameter was introduced as the first step on this path. It is an easy thing to do to route it to print output like this:
 

options nodate nonumber;
title1;
footnote1;

%lstattrib(sashelp.class,logfile=print);

The Next Step Down the Path of Programmatic Use

Hopefully you can see that the above way of using %lstattrib, i.e. writing the generated code to a temporary flat file and to "%include" it will run the code. This is true programmatic use of the macro. But it is a bit limited. Suppose we wanted to MERGE two or more datasets together instead of using the SET statement. We can not do this using the dsset= parameter. So there are more parameters to allow us to do this. We can use lenfile=, initfile= and attrfile= to receive the LENGTH code, the initialization code and attrfile will contain the code for cancelling the formats and informats right down to the final KEEP statement.

As for the input datasets and output dataset then we can code this ourselves. That way we are free from any limitations of the macro.

Note that in the code below we have no need to redirect the initialization code as we have set init=no. By setting dsset= to a null then we prevent the macro from writing the SET statement.
 

filename lengen  TEMP;
filename attrgen TEMP;
 

%lstattrib(sashelp.class,dsset=,init=no,
lenfile=lengen,attrfile=attrgen);
 

data myclass;
  %include lengen / source;
  set sashelp.class;
  %include attrgen / source;
run;
 

filename lengen  CLEAR;
filename attrgen CLEAR;

Fixing your dataset

If you have generated code for the sas lengths and attributes, as was done in the example in the previous section, then you could keep the code to apply a final fix to your data if you have been doing some manipulation of the dataset and you want to make sure that the characteristics of the dataset have not changed in any way. What follows is a real life example from a program I have been working on where I have to plug some missing values such as adding a sequence number and populating some variable values and I want to be 100% sure that the data is left in the correct structure. I have kept the generated code for lengths and attrributes and I reapply it to the dataset when I write it out to the target library.
 

        /***************************************
          Fix any problems due to data plugging
         ***************************************/
*- Reapply the correct lengths and variable order   -;
*- and reapply the correct attributes and KEEP list -;
*- so that any changes or additions due to data     -;
*- plugging are corrected. Data finally written to  -;
*- TARGET library at this point.                   -;
data TARGET.&target;
  %include lencode;
  SET &target;
  %include attrcode;
run;
 

Towards Being Metadata Driven

Up until now we have used the first positional parameter as the source of the correct dataset attributes. If you remember, that is how the macro first started out - as a tool to write the attributes of a correct dataset to the log where it could then be copied and emailed to programmers. But if we are going to use %lstattrib programmatically then there will be times that this correct dataset is not an actual dataset but rather a description of one that it held as data in a dataset (I am using the term "metadata" in the sense that it is not purely "data about data" but also includes instructions on what to do with data). The macro has a parameter named dsattr= where you can specify that it should use this dataset as the source of the attributes held as data in columns that are named "name, length, type, format, informat, label, varnum". The structure has to be the same as you would get from DICTIONARY.COLUMNS or SASHELP.VCOLUMN and not the structure you get from "proc contents".

The code below will use DICTIONARY.COLUMNS to give the structure of the SASHELP.CLASS dataset instead of using SASHELP.CLASS directly. Output will be written to the output window.

For this example I am asking for the initialization code and removing the SET statement by setting dsset= to null. If this code is run it will give me a one observation dataset with missing values that has the same structure as SASHELP.CLASS.
 

options nodate nonumber;
title1;
footnote1;

PROC SQL NOPRINT;
  create table classattr as
  select * from DICTIONARY.COLUMNS
  where libname="SASHELP" and memname="CLASS";
QUIT;
 

%lstattrib(dsattr=classattr,logfile=print,init=yes,
dsset=,dsout=myclass);

The Namepref= Parameter

There is one last parameter to tell you about before this User Guide is complete. That is the namepref= parameter where you can specify a prefix for all the variable names. The reason you might want to use this is where there is a possibility of assigning variables values where they are dependent on variables in an input dataset that might possibly have the same name. I will explain further for this very important topic:

Suppose you had gone down the "metadata driven" path and you are using data not purely in the sense of "data about data" but also including the sense of "instructions what to do with data". You have metadata to tell you what variables should be in what datasets and what their attributes should be. You also have metadata mapping information that tells you how to populate the variables in the dataset in terms of setting contents or deriving information from variables in an input dataset. Now suppose you were populating a character variable named "gender" based on a numeric "gender" variable in the input dataset then this would cause a syntax error. But by setting namepref= to a value you can avoid this situation. If you set namepref=NEW_ then if you had a gender variable in your input dataset then there would be no syntax error because you would be populating the variable NEW_GENDER with the value instead of GENDER. And at the end of the data step, only the NEW_ variables would be kept so the "gender" variable in the input dataset would be dropped. Problem solved, except now all your variables begin with "NEW_". But it is an easy thing to drop that prefix in a later data step for all the variables.

This topic is important enough for a thorough example on how to use it. Below is code from a previous example to create a MYCLASS dataset except now we will call it NEW_CLASS instead and prefix all the variables with NEW_ .
 

SASHELP.CLASS will be read but all the variable names will be prefixed with NEW_ in a "rename" clause using the macros %editlist and %varlist (that you can find on this web site).
 

%lstattrib will be used to list the attributes of NEW_class to the log.
 

The dataset NEW_class will be read in with a "rename" clause that drops the NEW_ prefix, again using the %editlist and %varlist macros (that you can find on this web site).
 

Finally, the attributes of the "class" dataset are listed to the log using %lstattrib.
 

If you run the code then you will see that both the New_class and Class datasets in the work library have the same contents as SASHELP.CLASS .
 

filename lengen  TEMP;
filename attrgen TEMP;
 

%lstattrib(sashelp.class,dsset=,init=no,
lenfile=lengen,attrfile=attrgen,namepref=NEW_);
 

*- This is a rename list that will be used for sashelp.class -;
*- to add a NEW_ prefix to the variable names. -;
%PUT >>>>>> %editlist(%varlist(sashelp.class),'&item=NEW_&item');
 

data NEW_class;
  %include lengen / source;
  set sashelp.class(rename=(%editlist(%varlist(sashelp.class),'&item=NEW_&item')));
  %include attrgen / source;
run;

%lstattrib(NEW_class,init=no,dsset=);
